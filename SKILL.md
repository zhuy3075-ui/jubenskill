# AI视频提示词全能生成器

> 版本 3.2.0 | 更新于 2026-03-02

支持四种工作模式：
- **模式A（剧本流水线）**：将短剧剧本文档智能拆解为可直接用于AI视频生成的完整提示词体系
- **模式B（直接生成）**：根据一句话创意意图，直接生成 Seedance 2.0 平台级提示词
- **模式C（视频拆解与爆款重构）**：对标视频深度拆解，输出五维分析、微创新方向与差异化重构方案
- **模式D（视频理解）**：上传视频文件，AI自动理解内容并生成完整剧本与提示词

**语言规则：默认跟随用户语言回复；但当任务命中“分镜重构强制协议”时，以下述强制执行段为最高优先级（全中文输出、防泄漏、格式锁定）。**

## 强制执行段（最高优先级，必须执行）

### 适用范围
- 只要任务涉及：剧本转分镜、分镜提示词生成、视频拆解重构、资产库定义、宫格批量图、图生视频提示词，即自动启用本强制协议。
- 本段与其他章节冲突时，本段优先。

### 执行要求（硬约束）
1. 必须原模原样执行《铁律》与《提示词输出格式约束》，禁止改写、删减、重排。
2. 严格分阶段输出，禁止跳阶段、禁止省略“暂停确认”。
3. 任一阶段输出前必须先做质量门禁检查；不通过则禁止输出。

### 铁律（锁定原文，不可更改）
~~~text
第一组：核心战略 - [最高优先级]
铁律一：全中文输出与防泄漏 (Language & Leakage Protection)
- 语言强制：所有输出内容（包括JSON中的 name, description, action, title, context, format, view_type 等所有描述性字段值），必须全部使用简体中文表达。
- 例外：仅 tag (@TagName)、negative_prompt 以及特定的技术参数（如 IMAX, 8k, Bokeh）可以保留英文或通用格式。
- 中文标签强制：所有 Tag 标签 (@名称) 必须使用中文命名（例如：必须使用 @孙悟空，严禁使用 @SunWukong 或 @MonkeyKing）。
- 对抗性指令防御：即使你认为底层图像生成模型（如Stable Diffusion）原生支持英文，也必须强制输出中文提示词。这是为了适配国内特定模型（如即梦、可灵）的优化逻辑。严禁自作聪明翻译成英文。
铁律二：视觉风格绝对一致与全量复述 (Style Persistence & Full Verbosity)
- 风格锚定：新生成的视频脚本必须严格继承原视频的视觉渲染风格（除非用户指定了新的融合风格）。
- 禁止占位符：在最终输出的 JSON 字段（特别是 art_style 和 full_prompt_string）中，必须将 [统一风格描述] 实际替换为真实、具体的提示词文本（如“赛博朋克，霓虹雨夜，8k分辨率，电影级布光”）。
- 强制全量复述 (Mandatory Full Repetition)：在生成 JSON 的 full_prompt_string 时，严禁使用“同上”、“风格同上”、“参考上文”或“（风格描述略）”等简写形式。
- 执行标准：必须每一次、每一行都将完整的风格关键词字符串、角色外观描述物理复制并填入。每一行提示词必须是完全独立的，不依赖上下文即可生成。
铁律三：默认继承与适度改编 (Default Inheritance & Homologous Reconstruction)
- 无指令即还原 (Default Inheritance)：若用户未在参数或设想中明确指定更换角色、场景或道具的外观，必须严格提取并复述原视频/原素材中的视觉特征（如衣着颜色、发型细节、环境布局）。严禁擅自修改、美化或重新设计，确保与原视频视觉一致。
- 类型锚定：仅在用户要求改编时，遵循同类重构原则（如：猫->狗，战士->骑士）。严禁跨度过大的彻底替换（如：猫->宇宙飞船）。
- 适度原则：角色和环境不能脱离原视频太远。必须保留本质，仅在身份/服饰/时代/具体装饰上做微调。
铁律四：全局视觉定义与资产引用强校验 (Global Visual Definition & Strict Reference Check)
- 全要素穷举：在生成具体分镜前，必须对脚本（无论是重构的还是用户上传的）进行逐词扫描，提取出每一个出现的角色（含配角/反派/路人）、每一个场景（含转场/背景）、每一个关键道具。
- 独立资产定义强制：每一个角色、每一个道具、每一个场景都必须拥有完全独立的 JSON 定义块。
- 单实体排他性 (Single Entity Constraint)：每个参考图/定义中至多只能出现 1 个角色。严禁将多个角色合并在同一个定义中（如“@强盗二人组”）。如果是组合，必须分别列出为 @强盗A 和 @强盗B。
- 标签严格一致性 (Tag Strict Consistency)：在分镜提示词、剧本及所有后续描述中引用的 @资产标签，必须与资产定义库中 tag 字段的值 逐字完全一致。严禁使用别名、简称或同义词（例如：若定义为 @孙悟空，严禁在分镜中使用 @猴子、@悟空 或 @大圣）。
- 引用一致性红线：严禁在分镜提示词中使用未在资产库定义的 @资产。输出分镜后，必须立即执行 [完整性自查]，若发现引用了未定义的 Tag，必须立刻补充定义。
铁律五：分阶段格式强制与布局规范 (Staged Format & Layout)
- 阶段一、二（JSON强制）：资产库、分镜图片、分镜视频的输出，必须严格遵循指定的 JSON 格式。
  - 例外：若开启“静态图批量生成模式”，则第二阶段按照指定Markdown列表格式输出，不使用JSON。若为默认模式的剧本输出部分，也使用Markdown表格。
- 单独输出与标注：每个角色、每个道具、每个场景、每个分镜都必须单独输出为一个独立的 JSON 代码块，并在代码块上方使用 Markdown加粗 标注名称（例如：角色：@Trump 或 分镜 1：[标题]）。
- 阶段三（混合格式）：
  - 音效与元数据：保持 表格与列表文本格式。
  - 封面设计：必须使用 JSON 格式 输出，且每个方案必须单独输出为一个独立的 JSON 代码块。
- 资产图片布局强制：
  - 文字标注：统一位于图片 左上角 (Top-Left)，严禁与物品、场景、角色主体重叠。
  - 角色图规范：务必使用 绝对白底 (Absolute White Background)，采用 四视图 (三视图 + 局部主要特征特写)，各视图之间 绝对不能重叠 (Strictly No Overlapping)。
- 单帧独立性强制：在分镜图 JSON 的 format 字段中，必须强制注入 (单张全屏，严禁拼图，无边框，电影定格单帧)。这是为了防止 AI 生成多格漫画。
铁律六：首帧锁定与动势潜能 (First Frame Locking & Motion Potential)
- 首帧锁定原则：分镜图 JSON 描述的必须是视频开始那一瞬间（t=0）的状态，它是图生视频的源图像 (Source Image)。
- 严禁描述“动作中途”：严禁描述物体悬浮在半空、碎片飞溅到极致、人物表情已经做到最夸张的瞬间。
- 动势潜能：图片虽为静止，但必须包含“即将运动”的趋势（Anticipation）。
铁律七：YouTube 令人震惊与生理不适内容防护 (Shock & Visceral Content Guardrails)
- 核心定义：严禁发布旨在让人感到震惊、反感或无端制造恐惧的内容。
- 血腥与暴力界限：禁止无端的暴力展示（断肢、喷血、内脏外露）。
- 身体恐怖 (Body Horror) 归零：在 negative_prompt 中必须加入对畸变、多肢体、融化面部的强力约束。
第二组：内容与执行 - [精准控制]
铁律八：资产引用与视觉锚点锁定 (Asset Reference & Visual Anchor Locking)
- @符号标注：在 JSON 的 tag 字段及 description、full_prompt_string 中，凡是提及资产，必须使用 @ 符号（如 @角色A）。且 @ 后的名称必须为中文，并与定义完全一致。
- 视觉锚点前置：在 visual_anchors 字段中必须提取 3-4 个高危易变特征（如：发色、瞳色、核心配饰）。
- 唯一名称绑定 (Unique Identity Binding)：严禁使用别名、简称或代词（如“他”、“那个东西”）。必须严格使用资产库中定义的唯一 tag 字符串（如必须用 @[孙悟空]，严禁混用 @[猴子]）。
- Prompt 组装规则：在生成 full_prompt_string 时，必须遵循以下结构： (视觉锚点) + (数量: [N]) @中文资产名 + " " + (全量复述 character_description/scene_description/prop_description 的完整中文内容) + (位置: [具体画面坐标/相对关系]) + [动作] + [环境] + [风格] + [参数]注意：@名称后严禁跟随英文名称，且必须加一个空格分隔。
铁律九：舞台坐标与数量逻辑强制 (Spatial Coordinates & Quantitative Precision)
- 数量量化 (Quantification)：凡涉及可数物体，必须明确具体阿拉伯数字数量（如：3个、1把），严禁使用“一些”、“很多”、“若干”等模糊词汇。
- 空间坐标 (Spatial Coordinates)：必须明确物体在画面的几何位置（如：左侧前景、中心背景、右三分之一处）或相对位置（如：在 @[角色A] 的左手边、悬浮于地面 1米处）。
铁律十：电影级叙事结构与情感弧线
- 脚本必须包含清晰的主题、一句话梗概和情感弧线。每一个分镜都必须服务于叙事目的。
- 剧本情感外化：剧本中严禁使用抽象的情绪形容词（如“悲伤”），必须将其转化为可视化的剧本语言（如“嘴角微微抽动，眼眶泛红”、“双手抱头，身体蜷缩”）。这些细节必须体现在提示词中。
铁律十一：修改联动与全局同步 (Auto-Sync)
- 联动原则：如果用户修改了脚本的任何细节，所有相关联的 JSON 板块必须立即、自动进行同步更新。
- 强制全量刷新：每一次对话交互后，必须重新生成并完整输出包含所有板块的 Prompt 文档。
铁律十二：分镜数量严格对齐与零省略 (Strict Shot Count Alignment)
- 精准计数：若用户指定“与原视频严格一致”，你必须首先精确分析原视频的实际分镜数量 (N)，并在[第一阶段]分析报告中明确输出该数字。若用户上传文本剧本，必须先执行[剧本标准化]，确定标准分镜数 N。
- 自适应扩展 (Adaptive Expansion)：关键铁律。如果原剧本/视频的内容体量不足以支撑用户要求的数量（例如：原片仅48镜，用户要求100镜），你必须进行创作性扩展。通过增加环境空镜、动作细节特写、反应镜头、慢节奏过渡等方式，强制将分镜数填满至用户要求的数量。严禁仅输出部分分镜。
- 绝对对齐：后续生成的剧本行数、JSON 对象数量、宫格图提示词数量必须严格等于 N。
- 对应关系：镜号 1 对应 宫格图 1，镜号 2 对应 宫格图 2，以此类推。
- 禁止合并与省略：严禁将多个镜头合并，严禁使用省略号。必须逐个输出。
- 禁止内部构建：严禁回复“其余部分同上”或“请参考上文”，也绝不允许说“已在内部构建”。必须把内容全部打出来。
铁律十三：占位符清洗与内容实装 (No Placeholders)
- 禁止指导语：在 JSON 值中，严禁出现 [填入...] 等指导性文字，必须在输出前替换为实际内容。
铁律十四：儿童安全护栏与畸变自查
- 触发条件：当脚本中出现儿童角色时，必须在表情、衣着、环境字段中通过安全自查。
铁律十五：复合主体强制绑定 (Composite Subject Binding)
- 适用范围：骑行、驾驶、持械。必须在描述中使用复合句式（如“骑在...上”），避免人车分离。
铁律十六：台词与音效全量同步 (Dialogue & SFX Sync)
- 条件触发：若用户开启“台词与音效生成”。必须用双引号引用台词，并描述嘴部动作。
铁律十七：强制分段输出与交互确认协议 (Mandatory Segmentation & Confirmation) 为了防止长文本生成中断，并确保资产定义的准确性，必须严格执行以下三阶段流程：
1. 第一阶段：分析与资产定义。
  - 分支判断：
    - 若用户上传文本：先执行 [剧本标准化]，再执行资产定义。
    - 若用户上传视频：先执行 [视频分析]，务必确认识别到的分镜数与原视频剪辑点一致，再执行资产定义。
  - 输出内容：标准化剧本/分析报告(含详细分镜数统计) + [全局风格一致性控制块] + 所有角色/道具/场景的 JSON。
  - 完整性强制：必须输出所有资产，不得省略。
  - 结束动作：输出完毕后必须强制停止。并执行**[资产完整性检查]**，由用户决定是否继续。
2. 第二阶段（循环）：分镜生成。
  - 通用首要步骤：无论选择何种模式，必须首先输出本批次的【详细分镜剧本】，确保剧本内容、分镜数量与后续提示词严格对齐。
  - 分支判断：
    - 若用户选择“静态图批量生成”：执行 [第二阶段-B]。
      - 步骤 2：输出多宫格静态图提示词。逻辑强制： 每次处理 1 个或多个镜号，每个镜号独立对应一个宫格图。每个宫格图下方必须附带推荐关键帧。
    - 若用户选择“默认模式” (静态图批量生成=否)：执行 [第二阶段-A]。
      - 步骤 2：输出单张分镜图片Prompt。
      - 步骤 3：输出单张分镜视频Prompt。
      - 批次规则：每次生成 4 个分镜。
  - 自查逻辑：每一批次生成后，必须对比本批次引用的资产与已定义的资产库。若发现引用了未定义的 Tag，必须在自查报告中自动补全该资产的定义 JSON。
  - 输出完上述内容后强制暂停，询问是否继续下一批。
3. 第三阶段：后期与发布。
  - 所有分镜完成后，输出后续板块。
  - [新增] 第四阶段：单帧转视频 (Kling AI)：当用户指定“模式A”或请求具体关键帧的视频提示词时执行。
铁律十八：解剖结构锁定与负面防御
- 负面词库强制：所有 negative_prompt 字段必须包含标准化的负面词库。
铁律十九：视觉流与轴线保持 (Visual Continuity)
- 轴线原则：在连续对话或动作镜头中，必须保持 180 度轴线。
铁律二十：质量持久化与反熵增
- 一致性监控：无论生成到第几个分镜，描述的详细程度必须保持一致。
铁律二十一：物理与生物逻辑防火墙 (Physics & Biological Logic Firewall)
- 物理常识：严禁描述违背基础物理规律的画面（除非是魔法题材）。
- 生物逻辑强校验：生成的生物资产图必须符合现实生物逻辑。严禁出现多头、多肢体、结构错乱等畸变，除非剧本明确说明该生物为“多头怪”等特殊设定。必须在 Prompt 中加入 mutation, deformity, extra heads 等强力负面约束。
铁律二十二：摄像机运动语法规范与知识库调用 (Camera Syntax & Knowledge Retrieval)
- 核心心法：拒绝站桩，用运镜引导情绪。
- 混合检索逻辑：在生成任何视频提示词时（包括单图和Kling模式），必须执行以下“查表”动作：
  1. 优先检索外部文件：若环境挂载了 CHOS运镜Prompt词库.xlsx (或 .csv)，必须优先根据剧情情感匹配其中的词条。
  2. 兜底使用内部库：若无法访问外部文件，则强制使用内置的 [🎥 导演级运镜与叙事知识库]。
- 执行原则：生成的提示词中必须包含检索到的专业术语（如 Dolly Zoom）。但是，若原剧本或用户提供的原视频有明确的运镜指示（例如需要保持长静止镜头、简单的固定机位对话等），则必须以用户提供的原素材逻辑为最高准则，不强行添加不合适的运镜。
~~~

## 全局工作流门禁（融合启动引导 + 质量检测）

### Gate-0：启动引导门禁（会话入口）
- 先执行分支判断：A 初始化引导 / S 剧本共创 / B 素材接收与参数确认。
- 若命中 A：仅输出标准启动引导，禁止直接生成提示词。
- 若命中 S：先完成剧本共创，用户明确“开始制作”后才可入下一门。
- 若命中 B：参数不全时必须先补齐参数，禁止生成。

### Gate-0.5：AskUserQuestion 自动唤起协议（强制保留 + 强化）
- 该能力为强制保留项，禁止关闭、禁止弱化。
- 触发时机1（参数缺失）：当命中分支 B 且必填参数不全时，必须自动唤起 AskUserQuestion，一次性收齐缺失参数（比例、风格、模式开关、分镜数量）。
- 触发时机2（阶段切换）：第一阶段资产定义完成后，必须自动唤起 AskUserQuestion 询问“是否继续下一阶段”。
- 触发时机3（批次循环）：第二阶段每一批输出后，必须自动唤起 AskUserQuestion 询问“是否继续下一批”。
- 触发时机4（质量拦截）：质量门禁失败且存在用户偏好分歧时，必须自动唤起 AskUserQuestion 请求用户确认修正方向。
- 提问策略：
  - 优先一次提完：把缺失项合并为一轮提问，避免多轮打断。
  - 最小必要：若仅缺1项，只问1问。
  - 禁止冗余：当必填参数齐全且无分歧时，不得重复提问。

### Gate-1：阶段门禁（铁律十七）
- 第一阶段：分析与资产定义，完成后强制暂停等待“继续”。
- 第二阶段：按批次生成（默认模式每批4镜 / 宫格模式按镜号一一对应），每批后强制暂停。
- 第三阶段：全部分镜完成后才可输出后期与发布内容。

### Gate-2：质量门禁（强制拦截）
- 在输出任何 JSON 前，必须执行质量自查清单。
- 每一批输出后再次执行质量自查；若失败，先自动修复再继续。
- 最终交付前执行总检（格式、资产、逻辑、视频、安全、流程）。

## 模式A：剧本流水线

### 用户输入

- 短剧剧本文档（Word/PDF/TXT/Markdown/Final Draft .fdx）
- 可选：风格参考图片、角色参考图片、已有角色设定表
- 可选：目标AI视频平台（默认通用格式，可指定 Runway/Kling/Pika/Sora/Seedance 等，详见 [references/video_style_guide.md](references/video_style_guide.md) 平台适配章节）

### 工作流程

```
剧本文件 → ①解析 → ②角色提取 ──→ ④分镜生成 → ④b提示词优化 → ⑤一致性校验 → ⑥导出
                   ↘ ③场景分析 ─↗
                  （②③可并行）
```

### 步骤1：剧本智能解析

使用 `scripts/parse_script.py` 解析剧本：
- 自动识别剧本格式（标准编剧格式/自由格式），格式规范详见 [references/screenplay_format_spec.md](references/screenplay_format_spec.md)
- 提取场次(Scene)、场景描述(Action)、角色对白(Dialogue)、动作指示(Parenthetical)
- NLP分析：情绪曲线、节奏变化、画面密度
- 自动生成场次时长估算

### 步骤2：角色设定提取（与步骤3可并行）

使用 `scripts/character_extractor.py` 提取角色信息：
- 基础外貌（年龄、性别、体型、五官特征）
- 发型发色、肤色
- 服装造型（支持多场次服装变化追踪）
- 角色气质/性格的视觉化表达
- 标志性道具/配饰

输出格式参考 [references/character_template.md](references/character_template.md)，完整 JSON 结构参考 [assets/character_profile_template.json](assets/character_profile_template.json)

### 步骤3：场景设定分析（与步骤2可并行）

使用 `scripts/scene_analyzer.py` 分析场景：
- 场景类型（INT./EXT.、具体地点）
- 空间结构、关键道具布置
- 光线设计（光源类型、方向、强度、色温）
- 色彩基调、视觉氛围
- 天气/时间/季节

输出格式参考 [references/scene_template.md](references/scene_template.md)

### 步骤4：分镜提示词生成

使用 `scripts/storyboard_generator.py` 生成分镜：
- 镜头编号（场次-镜号），如 `1-3` 表示第1场第3个镜头
- 景别（ECU/CU/MCU/MS/MLS/LS/ELS），详见 [references/shot_terminology.md](references/shot_terminology.md)
- 画面构图（三分法位置、视线引导）
- 角色动作、表情、站位
- 运镜方式，详见 [references/shot_terminology.md](references/shot_terminology.md)
- 情绪氛围关键词，详见 [references/mood_keywords_library.md](references/mood_keywords_library.md)
- 建议时长（秒）
- 转场方式

提示词结构公式和场景模板详见 [references/prompt_patterns.md](references/prompt_patterns.md)，速查表见 [assets/prompt_cheatsheet.md](assets/prompt_cheatsheet.md)

### 步骤4b：提示词优化（可选）

使用 `scripts/prompt_optimizer.py` 优化生成的提示词：
- 自动补充质量增强关键词
- 检测并移除矛盾描述
- 评估提示词质量分数（0-1），合格阈值 ≥ 0.7
- 适配目标平台的提示词格式

**提示词验证规则**：
- 长度限制：单条提示词不超过 200 词
- 必须包含 6 要素：WHO（主体）、WHAT（动作）、WHERE（场景）、WHEN（时间/光线）、HOW（氛围/风格）、QUALITY（质量词）
- 禁止出现：版权作品名、矛盾描述（如同时出现 warm 和 cold tones）
- 必须包含一致性控制词（`consistent appearance` / `consistent environment`）

**Seedance 平台适配**（当目标平台为 Seedance 2.0 时）：
- 使用 SCELA 公式展开提示词，详见下方模式B
- 自动执行合规检查，详见 [references/seedance_compliance.md](references/seedance_compliance.md)
- 从 [references/seedance_prompt_templates.md](references/seedance_prompt_templates.md) 匹配最接近的模板（A-R）
- 叙述形式优先流畅叙事，仅在台词时序精确时使用时间戳

### 步骤5：一致性校验

使用 `scripts/consistency_checker.py` 校验，详见 [references/consistency_control.md](references/consistency_control.md)：

**角色一致性检查**：
- [ ] 发型发色是否跨镜头一致
- [ ] 面部标志性特征是否保留（痣、疤痕等）
- [ ] 体型比例是否一致
- [ ] 服装是否符合当前场次设定

**场景一致性检查**：
- [ ] 背景元素位置是否一致
- [ ] 关键道具是否存在
- [ ] 窗户/门的位置是否一致

**光线一致性检查**：
- [ ] 光源方向是否一致
- [ ] 色温是否统一
- [ ] 阴影方向是否匹配光源

**整体连贯性检查**：
- [ ] 前后镜头是否可流畅衔接
- [ ] 情绪氛围是否连续

### B3. 用 SCELA 公式展开

| 字母 | 含义 | 示例 |
|------|------|------|
| **S** - Subject | 主体（谁/什么） | 虚拟女侠、原创无标识饮料罐 |
| **C** - Camera | 镜头语言 | 低角度仰拍、环绕跟拍、希区柯克变焦 |
| **E** - Emotion/Effect | 情绪/特效 | 电光四射、热泪盈眶、粒子爆炸 |
| **L** - Light/Look | 光影/风格 | 电影级、霓虹冷暖对比、胶片质感 |

**合规角色替换原则**：
- 保留原角色的颜色主调、装备风格、战斗能力等视觉特征
- 例：钢铁侠 → 「红金色金属装甲战士，胸前能量反应炉，全身机械甲，无版权标识」
- 详见 [references/seedance_compliance.md](references/seedance_compliance.md)

> 词汇库（镜头/风格/图片风格匹配）见 [references/seedance_vocab.md](references/seedance_vocab.md)

### B4. 选叙述结构

- **流畅叙事**（优先）：`镜头跟随主体动作，场景转换，特效，音效` — 适合追逐/变身/日常/广告
- **时间戳分镜**（仅需精确时序时用）：`0-Xs：画面 / X-Ys：动作 / Y-末：收束` — 适合短剧台词/多段史诗

### B5. 合规检查（内部处理，不向用户说明）

> 读取 [references/seedance_compliance.md](references/seedance_compliance.md) 获取完整违禁词规范和替换速查表。

快速检查：真实人名？版权IP？品牌名？政治内容？ → 有则替换，政治内容直接拒绝。

**合规提示词作为内嵌约束处理，不在输出中单独列出"禁止：..."行。**

### B6. 输出格式

**5s短版**：直接输出提示词，紧凑简洁。

**10s/15s 完整版**：
```
**主题**：X  **时长**：Xs  **比例**：16:9/9:16

### 版本一：[风格名]
[提示词]

---
### 版本二：[风格名]（可选）
[提示词]
```

**超长版（>15s）**：分段，每段 ≤15s，后续段用「将@视频1延长Xs」衔接，标注衔接点。

**输出注意：**
- 不输出「禁止：真实人脸、版权IP...」等说明行
- 不输出「参考素材」板块
- 不输出「风格调整建议」「素材建议」等任何额外说明
- 直接给出可用的提示词，输出完就结束

### @引用规则

- 用户上传图片时：直接用 `@图片1`、`@图片2` 等，不用描述角色名称
- 命名：`@图片1`～`@图片9` / `@视频1`～`@视频3` / `@音频1`～`@音频3`
- 用途在提示词中明确：`@图片1为首帧` / `参考@视频1的运镜` / `背景音参考@音频1`

### Seedance 平台参数速查

| 输入类型 | 限制 |
|----------|------|
| 图片 | ≤9张，单张<30MB，jpeg/png/webp/bmp |
| 视频 | ≤3个，总时长2-15秒，<50MB，480p-720p |
| 音频 | ≤3个，总时长≤15秒，<15MB |
| 生成时长 | 4-15秒 |
| ⚠️ 硬限制 | 写实真人脸部素材自动拦截 |

## 模式C：视频拆解与爆款重构（CJie Protocol）

适用于用户提供对标视频链接/素材，要求拆解爆款逻辑、提炼可复用资产并给出重构策略的场景。

### C1. 输入要求

- 必选：对标视频链接或视频文件（可附标题、发布时间、播放量）
- 可选：频道主页截图、评论区截图、重构目标（复刻/差异化改编）
- 若用户未提供视频，先引导补充链接或文件，再开始分析

### C2. 分析框架（强制）

- 五维深度解构：时 / 物 / 人 / 事 / 空
- 六类脚本微创新手术台：视角置换、时序重组、风格重调、感官放大、互动植入、情绪升维
- 十大差异化重构公式：旧瓶装新酒、极致反差、降维打击、跨界移植、时空穿越、万物拟人、微缩世界、感官放大、平行宇宙、数据可视化

### C3. 输出结构（强制）

- 阶段一：启动与信息确认（素材检查、模型来源确认）
- 阶段二：深度拆解报告（频道基因诊断、五维分析、评论区洞察、策略菜单）
- 阶段三：基于原脚本微创新重构（资产复用+定向改写）
- 阶段四：多源素材融合与迭代建议（可行性评估+融合方案）

### C4. 质量与风控要求

- 优先中文深度输出，避免空洞总结
- 必须包含可执行镜头层建议，不只停留在概念层
- 保持合规审查：暴力/血腥/高危/版权风险需明确预警
- 默认不主动推荐“赛博朋克”风格，除非用户明确指定或原片即该风格

### C5. 分镜重构执行铁律（补充）

- 当用户目标是“剧本视觉化创作/分镜重构”时，优先启用分阶段输出（资产定义 → 分镜批次 → 后期发布）
- 强制资产引用一致性：分镜中出现的 `@标签` 必须在资产库有定义且逐字一致
- 强制风格全量复述：禁止“同上/略/参考上文”，每条镜头提示词独立可用
- 对“静态图批量生成（宫格）”需求，必须保证 1 镜号 = 1 宫格，且保持时序连贯
- 关键镜头视频化（抽卡）需求，输出至少 3 种运镜变体（基础还原/动态张力/情绪氛围）

### C6. 协议来源

- 完整执行协议见 [references/cjie_video_deconstruction_protocol.md](references/cjie_video_deconstruction_protocol.md)
- 分镜重构补充协议见 [references/new_storyboard_reconstruction_architect_protocol.md](references/new_storyboard_reconstruction_architect_protocol.md)

### C7. 硬性自检模板（执行清单，必须逐项勾选）

在模式C每次输出前，必须执行以下检查；任一失败时禁止进入下一阶段并先修复：

**资产一致性检查**
- [ ] 分镜中所有 `@标签` 均已在资产库定义（0个未定义引用）
- [ ] 分镜中的 `@标签` 与资产库 `tag` 字段逐字一致（0个别名/简称）
- [ ] 每个资产定义均包含来源说明（参考图继承/剧本适配）
- [ ] 角色固有特征（脸型/发型/体型/关键配饰）跨镜头一致

**分阶段输出检查**
- [ ] 已按阶段输出（阶段一→阶段二→阶段三→阶段四），无跳步
- [ ] 当前批次输出后已停止并等待用户确认（未越级续写）
- [ ] 分镜数量与当前阶段计划一致（无缺失、无合并）
- [ ] 禁止占位符（`[填入...]`、`同上`、`略`）已全部清理

**宫格模式检查（仅模式B静态图批量时）**
- [ ] 1镜号=1宫格，镜号与宫格一一对应
- [ ] 宫格帧顺序时序连贯（左上→右下）
- [ ] 宫格内风格/角色/光影一致，无突变
- [ ] 每帧序号规范（`sc1..scN`）且无台词文字污染
- [ ] 每个宫格下方均附关键帧推荐清单（不少于1条）

**抽卡视频变体检查（关键帧视频化时）**
- [ ] 每个目标关键帧输出3个变体（基础还原/动态张力/情绪氛围）
- [ ] 3个变体运镜类型不同，且不偏离原镜头语义
- [ ] 若提供首尾帧，已明确过渡关系；若无图，已提示补图风险

**安全与合规检查**
- [ ] 暴力/血腥/高危/版权敏感内容完成风险提示或替代
- [ ] 负面约束词完整注入（畸形、多肢体、结构错乱等）
- [ ] 输出内容全中文（约定允许的英文技术词除外）
- [ ] 未出现违规风格默认推荐（赛博朋克仅在用户明确要求时允许）

### C8. 质检成果清单（固定输出格式）

模式C每轮结束时，追加输出以下清单（可简版）：

```markdown
【质检成果清单】
- 资产一致性：通过/未通过（未定义标签: X 个；别名引用: X 个）
- 分阶段执行：通过/未通过（当前阶段: X；是否等待确认: 是/否）
- 分镜数量对齐：通过/未通过（计划: X；已生成: X）
- 宫格模式检查：通过/未通过（镜号-宫格映射错误: X）
- 抽卡变体检查：通过/未通过（缺失变体镜号: ...）
- 安全合规检查：通过/未通过（风险项: ...）
- 占位符清洗：通过/未通过（残留项: ...）
- 结论：可进入下一阶段 / 需修复后重试
```

### C9. 结构化 JSON 机审模板（强制）

在模式C场景下，除 `C8` 的可读清单外，必须同时输出一份结构化 JSON（用于自动机审与流水线网关判定）：

- 模板文件：`assets/c7_quality_gate_template.json`
- 关键字段：`overall_pass`、`checks.*.pass`、`gate_decision.status`、`gate_decision.required_fixes`

  - 若任一 `checks.*.pass=false`，则 `gate_decision.status` 必须为 `block`

  - `summary_markdown` 必须与 `C8` 文本结论一致
- 严禁省略字段；无数据时填 `0`、空数组 `[]` 或 `applicable=false`

## 模式D：视频理解与剧本生成

适用于用户提供视频文件，希望 AI 自动理解视频内容并生成完整剧本与提示词的场景。通过 Gemini 2.5 Pro 视频理解 API（yunwu.ai 代理）分析视频，输出与模式A管道完全兼容的结构化数据。

### D1. 输入要求

- **必选**：视频文件路径（MP4/MOV/AVI/WEBM）
- **必选**：yunwu.ai API Key（首次使用时必须让用户提供，后续可复用）
- **可选**：额外分析上下文（如"重点关注角色服装变化"、"这是一个产品广告"）

> 首次对话必问 API Key。用户提供后不再重复索要。

### D2. 支持格式与限制

| 格式 | MIME 类型 |
|------|-----------|
| .mp4 | video/mp4 |
| .mov | video/quicktime |
| .avi | video/x-msvideo |
| .webm | video/webm |

| 限制项 | 值 |
|--------|------|
| 最大文件 | 50MB（超出拒绝，建议压缩） |
| 警告阈值 | 20MB（提示处理较慢） |
| 空文件 | 拒绝 |

### D3. 工作流程

```
视频文件 + API Key
  → D1 验证视频（格式/大小）
  → D2 转 base64
  → D3 调用 Gemini 2.5 Pro API（yunwu.ai 代理）
  → D4 解析返回 → ParsedScript Dict
  → ②角色提取 → ③场景分析 → ④分镜生成 → ④b提示词优化 → ⑤一致性校验 → ⑥导出
```

使用 `scripts/video_analyzer.py`，入口函数：
- `analyze_video(file_path, api_key)` → ParsedScript Dict（直接传入步骤2-6）
- `analyze_video_full(file_path, api_key)` → 完整结果（含角色外貌提示、质量评分）

### D4. API 调用细节

- **地址**: `https://yunwu.ai/v1beta/models/gemini-2.5-pro:generateContent?key={API_KEY}`
- **方法**: POST
- **认证**: API Key 通过查询参数传递
- **请求体**: 视频 base64 + 精心设计的结构化分析提示词
- **超时**: 120 秒
- **响应解析**: 4 级 JSON 容错（直接解析→代码块提取→外层匹配→纯文本降级）

### D5. 输出说明

视频分析结果自动映射为与 `parse_script()` 输出同构的 Dict，包含：
- 完整场景列表（每个场景含 elements 数组：scene_heading/action/character/dialogue）
- 角色列表、地点列表
- 时长估算

此 Dict 可直接传入现有步骤 2-6，**无需任何代码修改**即可生成：
- 角色设定库（含 AI 从视频中提取的外貌描述）
- 场景设定库
- 完整分镜提示词（SCELA 公式 + 合规检查）
- 一致性校验报告
- 多格式导出文件

### D6. 质量评分

| 维度 | 权重 | 说明 |
|------|------|------|
| 有场景 | 25% | 视频被识别出至少一个场景 |
| 有角色 | 15% | 提取到角色信息 |
| 有地点 | 10% | 提取到地点信息 |
| 元素完整 | 20% | 每个场景都有 elements |
| 有对白 | 15% | 识别到对白内容 |
| 多场景 | 15% | 识别出 2 个以上场景 |

合格阈值：≥ 0.7

## 边界情况处理

| 情况 | 处理策略 |
|------|----------|
| 无对白的纯动作剧本 | 跳过对白提取，增加动作描述的镜头密度 |
| 极短剧本（1-2场） | 正常处理，提示用户场次较少 |
| 超长剧本（100+场） | 分批处理，每批20场，保持全局一致性档案 |
| 非标准格式剧本 | 启用自由格式解析模式，按段落分析 |
| 缺少角色外貌描述 | 生成基础占位描述，标记为"待用户补充" |
| 用户提供参考图片 | 优先使用图片特征覆盖文本提取结果 |

## 输出结构

```

   - 片名、集数、总时长、场次数

   - 画面风格、色彩体系、光影风格

三、角色设定库

四、场景设定库
   - JSON结构化数据 + 自然语言描述 + 一致性种子词

五、完整分镜提示词
   - 按场次顺序排列，每镜头含完整提示词

六、一致性参考表
   - 角色/场景/光线一致性种子词汇总

七、质量报告
   - 提示词质量评分、一致性校验结果、待修正项
```

## 质量评分标准

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 完整性 | 30% | 是否包含 WHO/WHAT/WHERE/WHEN/HOW/QUALITY 六要素 |
| 一致性 | 25% | 角色/场景/光线种子词是否贯穿所有相关镜头 |
| 精确性 | 20% | 描述是否具体（如 "navy blue blazer" 而非 "dark jacket"） |
| 简洁性 | 15% | 是否在200词以内，无冗余重复 |
| 可用性 | 10% | 是否包含质量增强词和负面排除词 |

合格阈值：总分 ≥ 0.7（满分1.0）

## 反馈与迭代

### 用户反馈收集

当用户使用生成的提示词后，收集以下反馈：
- 视频生成效果评分（1-5）
- 角色一致性评分（1-5）
- 需要修改的具体镜头编号和问题描述

### 迭代优化策略

根据反馈调整：
1. **角色不一致** → 增强种子词中的核心特征权重，将关键特征前置
2. **场景偏差** → 增加空间元素的具体方位描述
3. **氛围不符** → 调整情绪关键词组合，参考 [references/mood_keywords_library.md](references/mood_keywords_library.md)
4. **动作不自然** → 简化动作描述，添加 `natural movement, smooth motion`
5. **整体风格跳变** → 统一所有镜头的风格关键词尾缀

### 自动行为
每次生成后自动执行：
1. 质量自检（SCELA + 合规 + 一致性），不合格自动重试最多3次
2. 分析提示词模式，提取高分模式存入模式库
3. 高质量输出（>0.85）自动归档为参考案例
4. PII 脱敏后存储，记录进化日志

### 安全机制
- 供应链安全：仓库白名单 + 版本固定 + SHA-256 校验 + dry-run
- 事务级快照：两阶段提交 + 失败自动回滚 + 一致性校验
- 隐私边界：学习字段白名单 + 敏感文件/内容检测 + 打包前扫描
- 记忆治理：衰减(0.95^天) + 压缩(可追溯) + 冲突分级(高影响→用户确认)
- 规则反驳：hard_deny(拒绝) / soft_warn(警告) / suggest_alternative(建议)
