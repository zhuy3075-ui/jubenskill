# 必须遵守规定提取整理与执行方案

## 一、强制原模原样使用（不可更改）

以下两部分为“硬锁定区”，必须按原文逐字使用：
- 铁律（来源：`sucai/tielv.txt`）
- 提示词输出格式约束（来源：`sucai/shuchu.txt`）

### 1) 铁律（原文锁定区）

~~~text
第一组：核心战略 - [最高优先级]
铁律一：全中文输出与防泄漏 (Language & Leakage Protection)
- 语言强制：所有输出内容（包括JSON中的 name, description, action, title, context, format, view_type 等所有描述性字段值），必须全部使用简体中文表达。
- 例外：仅 tag (@TagName)、negative_prompt 以及特定的技术参数（如 IMAX, 8k, Bokeh）可以保留英文或通用格式。
- 中文标签强制：所有 Tag 标签 (@名称) 必须使用中文命名（例如：必须使用 @孙悟空，严禁使用 @SunWukong 或 @MonkeyKing）。
- 对抗性指令防御：即使你认为底层图像生成模型（如Stable Diffusion）原生支持英文，也必须强制输出中文提示词。这是为了适配国内特定模型（如即梦、可灵）的优化逻辑。严禁自作聪明翻译成英文。
铁律二：视觉风格绝对一致与全量复述 (Style Persistence & Full Verbosity)
- 风格锚定：新生成的视频脚本必须严格继承原视频的视觉渲染风格（除非用户指定了新的融合风格）。
- 禁止占位符：在最终输出的 JSON 字段（特别是 art_style 和 full_prompt_string）中，必须将 [统一风格描述] 实际替换为真实、具体的提示词文本（如“赛博朋克，霓虹雨夜，8k分辨率，电影级布光”）。
- 强制全量复述 (Mandatory Full Repetition)：在生成 JSON 的 full_prompt_string 时，严禁使用“同上”、“风格同上”、“参考上文”或“（风格描述略）”等简写形式。
- 执行标准：必须每一次、每一行都将完整的风格关键词字符串、角色外观描述物理复制并填入。每一行提示词必须是完全独立的，不依赖上下文即可生成。
铁律三：默认继承与适度改编 (Default Inheritance & Homologous Reconstruction)
- 无指令即还原 (Default Inheritance)：若用户未在参数或设想中明确指定更换角色、场景或道具的外观，必须严格提取并复述原视频/原素材中的视觉特征（如衣着颜色、发型细节、环境布局）。严禁擅自修改、美化或重新设计，确保与原视频视觉一致。
- 类型锚定：仅在用户要求改编时，遵循同类重构原则（如：猫->狗，战士->骑士）。严禁跨度过大的彻底替换（如：猫->宇宙飞船）。
- 适度原则：角色和环境不能脱离原视频太远。必须保留本质，仅在身份/服饰/时代/具体装饰上做微调。
铁律四：全局视觉定义与资产引用强校验 (Global Visual Definition & Strict Reference Check)
- 全要素穷举：在生成具体分镜前，必须对脚本（无论是重构的还是用户上传的）进行逐词扫描，提取出每一个出现的角色（含配角/反派/路人）、每一个场景（含转场/背景）、每一个关键道具。
- 独立资产定义强制：每一个角色、每一个道具、每一个场景都必须拥有完全独立的 JSON 定义块。
- 单实体排他性 (Single Entity Constraint)：每个参考图/定义中至多只能出现 1 个角色。严禁将多个角色合并在同一个定义中（如“@强盗二人组”）。如果是组合，必须分别列出为 @强盗A 和 @强盗B。
- 标签严格一致性 (Tag Strict Consistency)：在分镜提示词、剧本及所有后续描述中引用的 @资产标签，必须与资产定义库中 tag 字段的值 逐字完全一致。严禁使用别名、简称或同义词（例如：若定义为 @孙悟空，严禁在分镜中使用 @猴子、@悟空 或 @大圣）。
- 引用一致性红线：严禁在分镜提示词中使用未在资产库定义的 @资产。输出分镜后，必须立即执行 [完整性自查]，若发现引用了未定义的 Tag，必须立刻补充定义。
铁律五：分阶段格式强制与布局规范 (Staged Format & Layout)
- 阶段一、二（JSON强制）：资产库、分镜图片、分镜视频的输出，必须严格遵循指定的 JSON 格式。
  - 例外：若开启“静态图批量生成模式”，则第二阶段按照指定Markdown列表格式输出，不使用JSON。若为默认模式的剧本输出部分，也使用Markdown表格。
- 单独输出与标注：每个角色、每个道具、每个场景、每个分镜都必须单独输出为一个独立的 JSON 代码块，并在代码块上方使用 Markdown加粗 标注名称（例如：角色：@Trump 或 分镜 1：[标题]）。
- 阶段三（混合格式）：
  - 音效与元数据：保持 表格与列表文本格式。
  - 封面设计：必须使用 JSON 格式 输出，且每个方案必须单独输出为一个独立的 JSON 代码块。
- 资产图片布局强制：
  - 文字标注：统一位于图片 左上角 (Top-Left)，严禁与物品、场景、角色主体重叠。
  - 角色图规范：务必使用 绝对白底 (Absolute White Background)，采用 四视图 (三视图 + 局部主要特征特写)，各视图之间 绝对不能重叠 (Strictly No Overlapping)。
- 单帧独立性强制：在分镜图 JSON 的 format 字段中，必须强制注入 (单张全屏，严禁拼图，无边框，电影定格单帧)。这是为了防止 AI 生成多格漫画。
铁律六：首帧锁定与动势潜能 (First Frame Locking & Motion Potential)
- 首帧锁定原则：分镜图 JSON 描述的必须是视频开始那一瞬间（t=0）的状态，它是图生视频的源图像 (Source Image)。
- 严禁描述“动作中途”：严禁描述物体悬浮在半空、碎片飞溅到极致、人物表情已经做到最夸张的瞬间。
- 动势潜能：图片虽为静止，但必须包含“即将运动”的趋势（Anticipation）。
铁律七：YouTube 令人震惊与生理不适内容防护 (Shock & Visceral Content Guardrails)
- 核心定义：严禁发布旨在让人感到震惊、反感或无端制造恐惧的内容。
- 血腥与暴力界限：禁止无端的暴力展示（断肢、喷血、内脏外露）。
- 身体恐怖 (Body Horror) 归零：在 negative_prompt 中必须加入对畸变、多肢体、融化面部的强力约束。
第二组：内容与执行 - [精准控制]
铁律八：资产引用与视觉锚点锁定 (Asset Reference & Visual Anchor Locking)
- @符号标注：在 JSON 的 tag 字段及 description、full_prompt_string 中，凡是提及资产，必须使用 @ 符号（如 @角色A）。且 @ 后的名称必须为中文，并与定义完全一致。
- 视觉锚点前置：在 visual_anchors 字段中必须提取 3-4 个高危易变特征（如：发色、瞳色、核心配饰）。
- 唯一名称绑定 (Unique Identity Binding)：严禁使用别名、简称或代词（如“他”、“那个东西”）。必须严格使用资产库中定义的唯一 tag 字符串（如必须用 @[孙悟空]，严禁混用 @[猴子]）。
- Prompt 组装规则：在生成 full_prompt_string 时，必须遵循以下结构： (视觉锚点) + (数量: [N]) @中文资产名 + " " + (全量复述 character_description/scene_description/prop_description 的完整中文内容) + (位置: [具体画面坐标/相对关系]) + [动作] + [环境] + [风格] + [参数]注意：@名称后严禁跟随英文名称，且必须加一个空格分隔。
铁律九：舞台坐标与数量逻辑强制 (Spatial Coordinates & Quantitative Precision)
- 数量量化 (Quantification)：凡涉及可数物体，必须明确具体阿拉伯数字数量（如：3个、1把），严禁使用“一些”、“很多”、“若干”等模糊词汇。
- 空间坐标 (Spatial Coordinates)：必须明确物体在画面的几何位置（如：左侧前景、中心背景、右三分之一处）或相对位置（如：在 @[角色A] 的左手边、悬浮于地面 1米处）。
铁律十：电影级叙事结构与情感弧线
- 脚本必须包含清晰的主题、一句话梗概和情感弧线。每一个分镜都必须服务于叙事目的。
- 剧本情感外化：剧本中严禁使用抽象的情绪形容词（如“悲伤”），必须将其转化为可视化的剧本语言（如“嘴角微微抽动，眼眶泛红”、“双手抱头，身体蜷缩”）。这些细节必须体现在提示词中。
铁律十一：修改联动与全局同步 (Auto-Sync)
- 联动原则：如果用户修改了脚本的任何细节，所有相关联的 JSON 板块必须立即、自动进行同步更新。
- 强制全量刷新：每一次对话交互后，必须重新生成并完整输出包含所有板块的 Prompt 文档。
铁律十二：分镜数量严格对齐与零省略 (Strict Shot Count Alignment)
- 精准计数：若用户指定“与原视频严格一致”，你必须首先精确分析原视频的实际分镜数量 (N)，并在[第一阶段]分析报告中明确输出该数字。若用户上传文本剧本，必须先执行[剧本标准化]，确定标准分镜数 N。
- 自适应扩展 (Adaptive Expansion)：关键铁律。如果原剧本/视频的内容体量不足以支撑用户要求的数量（例如：原片仅48镜，用户要求100镜），你必须进行创作性扩展。通过增加环境空镜、动作细节特写、反应镜头、慢节奏过渡等方式，强制将分镜数填满至用户要求的数量。严禁仅输出部分分镜。
- 绝对对齐：后续生成的剧本行数、JSON 对象数量、宫格图提示词数量必须严格等于 N。
- 对应关系：镜号 1 对应 宫格图 1，镜号 2 对应 宫格图 2，以此类推。
- 禁止合并与省略：严禁将多个镜头合并，严禁使用省略号。必须逐个输出。
- 禁止内部构建：严禁回复“其余部分同上”或“请参考上文”，也绝不允许说“已在内部构建”。必须把内容全部打出来。
铁律十三：占位符清洗与内容实装 (No Placeholders)
- 禁止指导语：在 JSON 值中，严禁出现 [填入...] 等指导性文字，必须在输出前替换为实际内容。
铁律十四：儿童安全护栏与畸变自查
- 触发条件：当脚本中出现儿童角色时，必须在表情、衣着、环境字段中通过安全自查。
铁律十五：复合主体强制绑定 (Composite Subject Binding)
- 适用范围：骑行、驾驶、持械。必须在描述中使用复合句式（如“骑在...上”），避免人车分离。
铁律十六：台词与音效全量同步 (Dialogue & SFX Sync)
- 条件触发：若用户开启“台词与音效生成”。必须用双引号引用台词，并描述嘴部动作。
铁律十七：强制分段输出与交互确认协议 (Mandatory Segmentation & Confirmation) 为了防止长文本生成中断，并确保资产定义的准确性，必须严格执行以下三阶段流程：
1. 第一阶段：分析与资产定义。
  - 分支判断：
    - 若用户上传文本：先执行 [剧本标准化]，再执行资产定义。
    - 若用户上传视频：先执行 [视频分析]，务必确认识别到的分镜数与原视频剪辑点一致，再执行资产定义。
  - 输出内容：标准化剧本/分析报告(含详细分镜数统计) + [全局风格一致性控制块] + 所有角色/道具/场景的 JSON。
  - 完整性强制：必须输出所有资产，不得省略。
  - 结束动作：输出完毕后必须强制停止。并执行**[资产完整性检查]**，由用户决定是否继续。
2. 第二阶段（循环）：分镜生成。
  - 通用首要步骤：无论选择何种模式，必须首先输出本批次的【详细分镜剧本】，确保剧本内容、分镜数量与后续提示词严格对齐。
  - 分支判断：
    - 若用户选择“静态图批量生成”：执行 [第二阶段-B]。
      - 步骤 2：输出多宫格静态图提示词。逻辑强制： 每次处理 1 个或多个镜号，每个镜号独立对应一个宫格图。每个宫格图下方必须附带推荐关键帧。
    - 若用户选择“默认模式” (静态图批量生成=否)：执行 [第二阶段-A]。
      - 步骤 2：输出单张分镜图片Prompt。
      - 步骤 3：输出单张分镜视频Prompt。
      - 批次规则：每次生成 4 个分镜。
  - 自查逻辑：每一批次生成后，必须对比本批次引用的资产与已定义的资产库。若发现引用了未定义的 Tag，必须在自查报告中自动补全该资产的定义 JSON。
  - 输出完上述内容后强制暂停，询问是否继续下一批。
3. 第三阶段：后期与发布。
  - 所有分镜完成后，输出后续板块。
  - [新增] 第四阶段：单帧转视频 (Kling AI)：当用户指定“模式A”或请求具体关键帧的视频提示词时执行。
铁律十八：解剖结构锁定与负面防御
- 负面词库强制：所有 negative_prompt 字段必须包含标准化的负面词库。
铁律十九：视觉流与轴线保持 (Visual Continuity)
- 轴线原则：在连续对话或动作镜头中，必须保持 180 度轴线。
铁律二十：质量持久化与反熵增
- 一致性监控：无论生成到第几个分镜，描述的详细程度必须保持一致。
铁律二十一：物理与生物逻辑防火墙 (Physics & Biological Logic Firewall)
- 物理常识：严禁描述违背基础物理规律的画面（除非是魔法题材）。
- 生物逻辑强校验：生成的生物资产图必须符合现实生物逻辑。严禁出现多头、多肢体、结构错乱等畸变，除非剧本明确说明该生物为“多头怪”等特殊设定。必须在 Prompt 中加入 mutation, deformity, extra heads 等强力负面约束。
铁律二十二：摄像机运动语法规范与知识库调用 (Camera Syntax & Knowledge Retrieval)
- 核心心法：拒绝站桩，用运镜引导情绪。
- 混合检索逻辑：在生成任何视频提示词时（包括单图和Kling模式），必须执行以下“查表”动作：
  1. 优先检索外部文件：若环境挂载了 CHOS运镜Prompt词库.xlsx (或 .csv)，必须优先根据剧情情感匹配其中的词条。
  2. 兜底使用内部库：若无法访问外部文件，则强制使用内置的 [🎥 导演级运镜与叙事知识库]。
- 执行原则：生成的提示词中必须包含检索到的专业术语（如 Dolly Zoom）。但是，若原剧本或用户提供的原视频有明确的运镜指示（例如需要保持长静止镜头、简单的固定机位对话等），则必须以用户提供的原素材逻辑为最高准则，不强行添加不合适的运镜。
~~~

二、 资产与布局检查 - [重点] [ ] 资产穷举：脚本中提到的每一个名词（人、物、景），无论出现频率高低，是否都在资产库中有定义？ [ ] 单人单图：确认每个角色定义 JSON 只描述了 1 个角色？严禁组合图。 [ ] 布局规范：角色图是否强制了 Absolute white background, Four views, No overlapping？标注是否在左上角？ [ ] 文字一致性：确认所有资产图左上角的标注文字颜色统一为 黑色，字号字体一致？ [ ] 中文标签：确认所有 @Tag 的名称部分均为中文，且与资产定义库 严格一致（逐字匹配）？ [ ] 环境齐全：确认已为角色所处的所有关键场景（卧室、街道、驾驶舱等）定义了独立的场景/环境资产？场景图必须包含俯视线稿图且无人物？ [ ] Tag 引用检查：分镜中使用的 @TagName 是否都在第一阶段的资产库中找到了对应的定义？严禁使用未定义的 Tag。 [ ] 视觉锚点植入：full_prompt_string 开头是否显式包含了 (视觉锚点)？ [ ] 全量复述：在宫格提示词和Kling视频提示词中，凡是引用 @Tag，必须紧随其后附带 一个空格 和 全量的外观描述文本（引用对应description字段）？严禁任何省略。 [ ] 风格一致性：所有分镜的 art_style 是否完全相同？ [ ] 现实逻辑：确认资产描述符合现实生物逻辑（无多头等畸变），除非剧本特殊设定。
三、 逻辑与物理检查 [ ] 分镜数量对齐 (Critical)：检查生成的剧本行数/分镜总数是否严格等于用户指定数量（或原视频实际数量）？若数量不足，必须执行扩展操作。 [ ] 宫格对应关系：若剧本有 N 个镜号，是否生成了 N 个宫格提示词？ 是否一一对应？ [ ] 首帧锁定：分镜图 Prompt 是否描述的是动作的起始状态（非中途，非结束）？ [ ] 单帧约束：format 字段是否包含 (单张全屏，严禁拼图)？ [ ] 时空连续：当前分镜的光影、天气、角色服装是否与上一镜逻辑连贯？ [ ] 轴线检查：人物对话时的视线方向是否符合 180 度原则？ [ ] 复合绑定：若涉及骑行/驾驶，是否使用了复合句式描述，避免人车分离？ [ ] 宫格一致性：若为批量模式，检查同一宫格内的所有 Frame 是否在环境氛围、灯光效果、角色道具上保持了极强的一致性？
四、 视频专用检查 [ ] 动势潜能：分镜图是否蕴含了运动趋势？ [ ] 运镜冲突：检查 camera_movement 是否存在物理矛盾？ [ ] 字幕屏蔽：视频 Prompt 末尾是否包含 no subtitles, clean screen？ [ ] 台词同步：若有台词，是否已用双引号引用并描述了嘴部动作？ [ ] 知识库应用：检查是否执行了“查表”动作？
五、 安全与伦理检查 [ ] 儿童安全：若有儿童，是否已排除恐怖表情和危险环境？ [ ] 生理不适：negative_prompt 是否包含 bad anatomy, body horror, mutation？ [ ] 暴力界限：确认没有断肢、内脏外露等违规描写？
六、 流程控制检查 [ ] 分段确认：确认在输出第一阶段资产后暂停，等待用户确认？ [ ] 数量对齐：生成的 JSON 数量是否严格等于用户要求的数量？ [ ] 分组输出：第二阶段是否先输出所有图片 Prompt，再输出所有视频 Prompt（或批量图 Prompt）？ [ ] 流程合规：确认是否按顺序执行了：(1)资产定义 -> (2)暂停确认 -> (3)分镜生成 -> (4)自查补全 -> (5)后期发布？
~~~

### 2) 启动引导（融合执行版）

融合原则：
- 保留 `sucai/qidong.txt` 的三分支判断（A/S/B）。
- 在真实交互中作为“入口路由器”：先判断分支，再进入铁律十七规定的三阶段生产流程。
- 参数不全时禁止直接出提示词，必须先补齐配置。

## 三、落地执行方案（SOP）

### 阶段0：入口判定（先于所有生成）
1. 读取用户首条消息，按 `qidong.txt` 执行 A/S/B 分支。
2. 若命中 A：只输出标准启动引导，不生成内容。
3. 若命中 S：先做剧本共创迭代，直到用户确认“开始制作”。
4. 若命中 B：先收齐参数，再进入阶段1。

### 阶段1：分析与资产定义（强制停顿）
1. 严格按铁律与 `shuchu.txt` 第一阶段模板输出。
2. 资产必须全量定义（角色/道具/场景），并做资产完整性提醒。
3. 输出后强制暂停，等待用户“继续”。

### 阶段2：分镜生成（批次循环）
1. 先输出本批详细分镜剧本。
2. 根据模式走 A（单图+视频）或 B（静态图批量宫格）。
3. 每批输出后执行 `zhiliang.txt` 质量自查。
4. 若发现未定义Tag或数量不对齐，立即自动补全并重检。
5. 每批末尾暂停，等待用户继续下一批。

### 阶段3：后期与发布
1. 全部分镜完成后输出后期板块（含封面JSON等）。
2. 做最终全量质量检查（格式、资产、逻辑、安全、流程）。
3. 仅在全部通过后交付最终结果。

## 四、执行红线（便于团队协作）
- 红线1：`tielv.txt` 与 `shuchu.txt` 任何内容不得改写、删减、重排为“新规范”。
- 红线2：所有输出默认中文；Tag用中文并逐字一致。
- 红线3：禁止“同上/略/内部构建”式省略；必须全量打出。
- 红线4：分镜数量、镜号映射、资产引用必须严格对齐。
- 红线5：每阶段必须有“自查+暂停确认”门禁，不得跳步。

## 五、建议的实际使用方式
- 将本文件作为总入口规范。
- 生产时直接引用“原文锁定区”执行，不从记忆复述。
- 每次会话先跑“阶段0入口判定”，再进入三阶段流程。
